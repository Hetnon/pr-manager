# BoilerPlateReadMe

**This file is shared boilerplate metadata, not project documentation.**

It describes conventions that apply to **every repo derived from the
`express-react-appEngine-boilerPlate`** boilerplate. The canonical copy lives
in that boilerplate. When the boilerplate changes, every derived repo should
get the updated copy of this file too.

## Purpose

Three things, in order of importance:

1. **Identify a repo as boilerplate-derived.** If you see this file in a repo,
   you can assume the layout, server/UI conventions, deploy pipeline, and
   tooling come from the boilerplate. Use the boilerplate's docs as the source
   of truth for anything not project-specific.
2. **Record cross-repo conventions** that no single repo can enforce on its
   own (e.g. "do not edit this file at the project level").
3. **Track known gaps** in the boilerplate so derived repos inherit the same
   warnings and to-do list.

## Do not edit at the project level

The body of this file is meant to stay identical across all derived repos. If
you need to record something project-specific, put it in that repo's
`README.md`, not here. We can't technically enforce this — it's a convention
backed by the practice below.

## Cross-repo change propagation

When making a structural change to the boilerplate (build pipeline, deploy
config, shared conventions, security defaults, etc.), the change is not "done"
until every locally-known derived repo has been considered. **Before closing
the task, scan `D:\AI Projects\` for every folder containing this file** —
that's the live inventory, regardless of what's listed below.

As of this writing, the derived repos are:

- `D:\AI Projects\express-react-appEngine-boilerPlate\` — the boilerplate
  itself (canonical).
- `D:\AI Projects\agentic_enterprise\` — derived.
- `D:\AI Projects\pr-matrix\` — derived. Adds isomorphic-git + MUI on top of
  the base boilerplate; webpack config has an extra `ProvidePlugin(Buffer)`.

The procedure for any structural change:

1. Make the change in the boilerplate first.
2. **List every local copy** of a boilerplate-derived repo (search for this
   file under `D:\AI Projects\`). Don't rely on memory — repos get added.
3. For each derived repo, ask whether the change should be applied there now,
   then either apply it or leave a TODO entry in that repo's
   `BoilerPlateReadMe.md` (under **Known gaps**) explaining why it was
   skipped.
4. Update this file's `Known gaps` section in the boilerplate, then copy the
   file to every derived repo so they all see the same list.

The rule is: **a structural change is not complete until propagation has been
explicitly decided for every derived repo, even if the decision is "not now."**

## Shared conventions

Things every derived repo should follow.

### Shared Firestore emulator across repos

In dev, every boilerplate-derived repo points at the same Firestore emulator
on `localhost:8080` and the same shared snapshot folder
(`~/.firestore-emulator-data/` by default, override with
`FIRESTORE_EMULATOR_DATA_DIR`). Multiple repos can run concurrently:

- The first one to start spawns the emulator with `--import` and
  `--export-on-exit` pointed at the shared folder. Subsequent repos find the
  emulator already running on `:8080` and just connect.
- Each repo sets a **unique** `GOOGLE_CLOUD_PROJECT` in its `server/.env.dev`
  (convention: `dev-<gcp-project-id>`). The emulator namespaces collections
  by that project ID, so data is fully isolated between repos inside the
  single shared snapshot.
- Seed data persists across dev sessions because the snapshot is shared and
  doesn't live inside any repo. Graceful emulator shutdown (Ctrl-C in its
  terminal window) triggers `--export-on-exit`; a 5-min periodic export
  bounds data loss from ungraceful kills.

The bootstrap script (`scripts/gcp-bootstrap.js`) now writes
`GOOGLE_CLOUD_PROJECT=dev-${projectId}` so new derived repos get a unique
value automatically. Existing `.env.dev` files predating this change may
still say `demo-project` — fix them by hand.

## Known gaps

Things the boilerplate doesn't currently solve. Carried forward into every
derived repo by virtue of being in this file.

### App Engine Standard does not support websockets

The boilerplate ships with Socket.IO wired into the Express server and the
React client, and both `server/app.yaml` and `ui/app.yaml` declare
`env: standard`. **Socket.IO will not work on App Engine Standard** — Standard
does not support websockets and silently strips the upgrade. If a derived
project needs the realtime channel, switch the server's `app.yaml` to
`env: flex` (or migrate the server to Cloud Run), and re-test session
stickiness. The UI service can stay on Standard because it's static-files
only.

### resumeAutomation alignment

`D:\AI Projects\resumeAutomation_win11_laptop\webClient\` predates the
boilerplate and has drifted significantly: separate `dist/` output, no shared
`TypesAndInterfaces`, no KMS-envelope token storage, no Firestore-backed
sessions, different webpack/dev-server setup, different folder structure.
Bringing it onto the boilerplate is a planned future task — not blocked by
anything specific, just large. Until then, **do not assume changes to the
boilerplate apply there**.

## When to update this file

- A new derived repo is created → add it to the propagation list above.
- A new known gap is discovered → add it under **Known gaps** and copy the
  file to all derived repos.
- A known gap is resolved → remove it.
- Anything project-specific → **do not put it here**, put it in the project's
  own `README.md`.
